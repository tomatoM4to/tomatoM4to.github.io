# Database 정의
* 통합된 데이터(Integrated Data): 중복 배재
* 저장된 데이터(Stored Data): HDD, SSD
* 운영 데이터(Operational Data): 필요한 데이터여야 함
* 공용 데이터(Public Data): 공유 가능한 데이터

# DBMS 필수 기능
* 정의, 조작, 제어

# 데이터 독립성
* 논리적 독립성: 응용프로그램과 DB를 독립시킴
* 물리적 독립성: 응용프로그램과 저장장치를 독립시킴


# 스키마
**DB의 구조와 제약조건에 대한 전반적인 명세**
* 외부 스키마: 사용자의 입장에서의 데이터베이스의 논리적 구조
* 개념 스키마: 전체 데이터베이스의 논리적 구조
* 내부 스키마: 데이터베이스의 물리적 저장장치 입장에서 본 DB의 구조

***

# 설계 고려사항
* 무결성: 삽입, 삭제 갱신등의 연산 후에도 제약 조건을 만족
* 일관성: 데이터베이스 내의 데이터가 일관된 상태로 유지
* 회복: 시스템 장애로 손상된 데이터를 복구 가능
* 보안: 불법적인 접근으로부터 데이터 보호
* 효율성: 시스템의 성능을 최적화
* 확장성: 시스템의 확장이 용이

# 설계 순서
1. 요구 조건 분석
2. 개념적 설계: 정보 모델링, 개념스키마, E-R 다이어그램
3. 논리적 설계: DBMS에 맞는 논리 스키마 설계
4. 물리적 설계: 물리족 구조의 데이터로 변환
5. 구현: DDL로 DB 생성

# 무결성 제약조건
* Domain Integrity: 속성의 값은 특정 도메인에 속해야 함, 학번은 정수, 이름은 문자열
* Entity Integrity: 테이블의 기본키를 지정해야 함
* Referential Integrity: 외래키가 참조하는 테이블의 기본키와 타입이 같아야 함


***
# 트랜잭션
* 원자성: 커밋돼는 Rollback 되든, 관련기능: 회복
* 일관성: 성공적으로 완료 되더라고 규칙 준수, 관련기능: 무결성, 동시성 제어
* 독립성: 다른 트랜잭션에 영향을 받지 않아야 함, 관련기능: 동시성 제어
* 지속성: 성공적으로 완료된 트랜잭션은 영구적으로 반영되어야 함, 관련기능: 회복

# 정규화
* 제1정규형: 모든 속성이 원자값이여야 함
    * 같은 속성 내에 중복된 데이터가 없어야 함
* 제2정규형: 부분적 함수 종속이 없어야 함
    * 기본키(여기선 복합키)의 일부 속성이 다른 속성을 결정하지 않아야 함, 테이블 분리
* 제3정규형: 이행적 함수 종속이 없어야 함
    * 한 테이블에서 (학번 -> 학과 -> 학과 사무실) 처럼 비효율이 발생할 시, 테으불 분리
* BCNF: 결정자이면서 후보키가 아닌것을 제거
    * 결정자 X -> Y,중 X, 이떄 이 결정자가 후보키여야함
* 제4정규형: 다치 종속이 없어야 함
    * 이렇게 분리된 테이블중 중복된 데이터 최소화
* 제5정규형: 조인 종속성 이용
    * 위에서 분해한 테이블을 JOIN할시 다시 원래대로 복구되어야 함
