이전의 SMTP 나 Http, DNS 는 Server-Client 아키텍처 모델이였다. 가장 기본적은 가정은 서버가 해야하는 역할과 클라이언트의 역할이 나누어져 있고, 서버는 요청한 서비스를 수행한다.

서버가 request 가 많이 몰리게 되면 부하가 걸릴 수 있는데, 이럴떈 분산 시스템으로 해결할 수도 있다, 대표적으로 DNS 의 Top leven, Authoritative name server 등등으로 나누는 것

## P2P

항상 running 하는 서버가 없다. 스패셜한 노드가 없다. 모든 참가하는 디바이스가 서버이자 클라이언트가 되는 모델

내가 요청도 하지만 내가 가지고 있는 자원(파일) 을 제공하는 역할도 한다.

P2P 에 참여하는 디바이스는 Peer 라고 부른다.

대표적인 서비스론 `bitTorrent` 가 있다. P2P 의 가장 난해한 경우는 항상 attach 되어 있는 서버가 없고 기존에 attach 되어 있더라도 언제든 끊어질수 있다는점

받고있는 사용자 입장에서 **어디에 가서 받아야 하지? 를 처리하는점**

* 장점:
  * 가장큰 기대할 수 있는 장점은, 더 효율적으로 서비스, 자원을 distribute 할수 있다는점, 상식적으로 당연함
  * 얼마나 더 효율적일까? 에 대해 수식적으로 이야기 할 수 있어야 함
    * 세팅은 최초로 파일을 서비스 하는 서버가 존재함, 해당 서버로부터 데이터를 받고싶어하는 클라이언트가 여러개 있음, 피어들 간의 컨텐츠가 없을 수 있다 가정
    * 피어들이 받아온 컨텐츠를 서로 공유(서버의 역할)
    * file size 는 F 라 가정
    * u 는 uplink: 어떤 피어로부터 인터넷으로 올려보내는(업로드)
    * d 는 downlink: 어떤 피어로부터 인터넷으로 내려받는(다운)
    * u와 d는 링크의 커패시티라고 할 수도 있음
    * 서버로부터 F 가 인터넷에 업링크 되는 시간, 네트워크로 내보내는 시간 = `F / u`, N 명의 사용자가 있으면 `N * F / u` 가 됨
    * 해당 F 에 대해 downlink 되는 시간도 있음 `F / d`, `d` 는 각각의 사용자마다 다를 수 있음
    * 러프하게 생각하면 총 걸리는 시간은 `D >= max(N * F / u, F / d)` 가 될 수 있음, 왜냐면 `d` 가 매우 작은 사용자도 있을 수 있기 때문, 이를 `d_min` 이라고 표현할 수 있음, `D >= max(N * F / u, F / d_min)` 가 될 수 있음, 즉 `d` 가 가장 작은 사용자인 `d_min` 와 업링크 되는 속도보단 느리게 될것 이란 예기
  * p2p 로 넘어가면?
    * 서버는 똑같음 `F / u` 시간 걸림
    * 사용자도 `F / d_min` 시간 걸린다고 예기 함
    * 여기서부터 차이가 생김, `D >= max(F / u, F / d_min, NF / u1 + u2 + ... + ui)` 가 됨 업로드 하는 사용자가 많아질수록 더 효율적이란 뜻, `u1 + u2 + ... + ui` 는 업로드 하는 사용자들의 업링크 속도의 합

결국 장점은 Scale ability, 즉 사용자 수가 많아질수록 더 효율적이란 뜻

## Bit Torrent
tracker 와 torrent 라는게 새로 등장함

* torrent: 파일을 chunk 라는 작은 조각으로 나누어서 관리하는 파일?
* tracker: torrent 파일을 관리하는 서버, peer 들이 torrent 파일을 가지고 tracker 에 접속해서 다른 peer 들의 정보를 얻어오는 역할

Peer 가 처음으로 torrent 에 접속하면 나의 존재를 tracker 에게 알림, tracker 는 peer 들의 정보를 줌, 해당 파일을 가지고 있는, 전부 다주는건 아니고 필요한 만큼만 줌

새로 join 한 peer 는 해당 peer 들에게 connection 을 시도하게 됨, connection 을 할때도 chunk 단위로 요청하게 됨, 어떤놈한텐 1번chunk 어떤놈은 2번 chunk 요청하는 식

근데 갑자기 peer 들이 갑자기 끊기면?

만약 모든 file 을 다 받으면 떠나거나 아니면 계속 공유하거나 하는 선택 사항이 있음

* 보안 이슈가 생김: 포이즌 데이터 같은 data 자체가 변질된거 가지고 서비스 하는 이슈가 생길수도, 사용자가 chunk 를 제공 하는데 이름은 똑같은데 내용물이 오염됨 이런식.. chunk 마다 시그니처를 줄 수도 있음, 중요한 파일은 매 chunk 마다 시그니처를 주지만, 내용물 자체도 커지고 보안을 위한 추가적인 데이터다 보니 오버헤드처럼 느껴질 수 도 있음, 그리고 인증 관련 시스템도 따로 마련되어 있어야 함, 바이너리 청크마다 해시를 돌리는것도 있음
* 프리라이더 문제: 매우매우 심각한 문제임, 사람들이 가장 신경쓴 문제기도 함, 사람들이 공유를 잘 안해줌;; 다음에 참여할때 패널티 부여하거나 공유 한만큼 다운로드가 빨라지도록 할 수 있음, bittorrent 는 tit-for-tat 라는 전략을 사용함, Alice 는 4명에게만 줌 이때 4명은 가장 많이 준놈 기준임 싸가지 없는 애들한텐 chunk 주지 않음, 옵션적으로 unchoke 라고 랜덤하게 어떤 peer 에게 주기도 함, 그래서 Alice 는 상위 4명과 랜덤 1명 총 5명에게 줌, 이러다 보면 어느 순간 4명의 우선순위가 바뀌기도 함

> BitTorrent 는 완벽한 P2P 라고 보기 어려움.. Peer 들만 존재하는게 아니라 tracker 라는게 존재함, tracker 가 peer 들의 정보를 추적하고 있기 때문, 새로운 peer 가 등장하면 tracker 에게 등록을 하게 됨

> 초기 P2P 는 이렇게 누군가 모니터링 하고 관리하는 방식은 P2P 로 인정하지 않으려는 움직임도 있었음, 하지만 Server 가 없으면 network 을 찾아내야 하는 overhead 를 무시하기 어려워 계속 저울질 하다 현재는 이 방식도 인정하게 됨

## Distributed Hash Table
P2P 의 가장 큰 문제는 내가 원하는 파일이 어디에 있지..? 이 파일을 제공하는 Node 를 어떻게 알지? 가 핵심적인 문제임

BitTorrent 는 Server 가 해당 기능을 제공했지만 만약 Server 가 없다면 어떻게 해당 Directory 서비스를 어떻게 제공하냐, 대표적인 방식이 DHT 임

DHT distributed P2P database 라고도 불림

peer 들은 key-value 들을 가지고 있는데 해당 key 를 DHT 에 요청을 날림, 어디가 DHT 임? 그냥 날리는거임 Network 가 아랑서 찾아주게 설계함

새로운 peer 가 존재하고 파일을 공유하려는데, 그러면 DHT 에다가 등록하는 과정을 거침

## How to assign keys to peers?
근데 어떤 특정 정보를 누구한테 Assigning 하지? 가 중요한 문제 (key, value) pair 들을 p2p 에 조인하고 있는 노드들 (DHT 를 구성하고 있는 노드) 들이 나눠가지고 있는데.. 이중 누구에게 받아야 하는가? 가 문제임

Basic idea 는 다음과 같음
1. Convert each key to an integer
2. Assign integer to each peer, 만약 A 라는 파일을 가지고 싶으면 예를들어 파일 이름을 변환 시킴(해시 사용) 그리고 각각 참여하고 있는 노드들의 아이디도 변환 시킴 이렇게 두개의 integer 가 생김, 이게 같은 Node 가 파일 정보를 가질 수 있게 만든것이 핵심, 근데 만약 틀리면? 이럼 가장 가까운 노드가 제공하도록.. 만든.. 룰.. 시발 복잡하네

> Overlay network: 논리적 네트웤, 피지컬 위에서 동작

해시값이 무한정 커질순 없으니 한계를 정하는데 n bits 로 표현함, 결국 가질수 있는 개수가 [0, 2^n - 1] 개가 됨

해시값이 틀리면 (애초에 없었거나, 컴퓨터를 껐거나) 이럴경우 가장 가까운 노드가 제공하도록 룰이 만들어짐. 5번이 없음 6번이 서비스 하도록, 이건 서비스를 만드는 사람이 룰을 정할 수 있음

근데 부여된 ID 가 가깝다는건 실제 물리적으로 가까운게 아님, 어떻게 준거에 따라 달라질 수 있지만, 실제로 vpn 이나 p2p 나 logical network 은 이런걸 의미함

## file sharing example
분산된 DB 에 `yonna.mp3` 는 누가 가지고 있나? p2p 네트워크에 물어봐야함, 일단 `h("yonna.mp3")` 를 돌려봒더니 `h("114.70.235.211")` 에 대응이 된걸 확인함, 즉 1번 이란 친구가 찾고자 하는 파일에 대한 정보를 가지고 있음, 1번은 `143.248.9.1` 이 가지고 있다는걸 알고 있고 다운로드자는 해당 ip 에 connection 을 시도함

이렇듯 내가 원하는 컨텐츠의 해시값과 같은 결과를 내는 노드의 해시값을 비교

## 결과를 내는 노드는 어떻게 찾는가?
Circular DHT 방식으로 찾아감, 바로 IP 주소를 알아낼순 없으니 overlay network 에 던짐 `1110` 이 누구야? 계속 넘어가다 `1111` 이 감, `1110` 은 없으니 이놈이 서비스를 시작함

순차 탐색이니 효율적으로 찾을수 있는 방법이 연구되었고 완료됨
