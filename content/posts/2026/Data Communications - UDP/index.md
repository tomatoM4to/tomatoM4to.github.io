## Multiplexing & Demultiplexing
> 데이터를 송수신하는 프로세스(애플리케이션)를 구분하기 위해 사용

* Multiplexing(송신자 측): 컴퓨터 내에서 실행 중인 여러 개의 애플리케이션(소켓)에서 생성된 데이터를 하나의 네트워크 데이터로 묶어서 하위 계층으로 전달
* Demultiplexing(수신자 측): 네트워크를 통해 수신된 데이터를 헤더(Header)에 있는 포트 번호 등을 확인하여, 해당 컴퓨터의 적절한 애플리케이션 프로세스로 분배하는 과정

## UDP

> Transport Layer 에서 동작하는 Protocol 로, 빠르고 단순한 전송을 목적으로 한다.

* **Best-Effort**: 부가기능이 없는 아주 단순한 형태의 protocol
* **비신뢰성 및 비순서 전송**: 데이터(Segment)가 손실될 수 있으며, 손실되더라도 이를 복구하기 위한 추가적인 조치를 취하지 않는다. 또한, 목적지 까지 데이터가 순서대로 도착하는 것을 보장하지 않는다.
* **Connectionless**: 송신자와 수신자 간에 연결을 설정하는 handshaking 과정이 없다, 따라서 각각의 UDP Segment 는 서로 독립적으로 처리된다.

### UDP 를 사용하는 이유
> 속도와 단순함의 장점이 손실 위험을 감수할 수 있을때

* **연결 설정 지연 없음**: 통신 전 연결을 맺는 과정이 없으므로, 이로 인해 발생하는 RTT를 절약할 수 있다.
* **단순한 구조**: 송신자와 수신자 양측에서 연결 상태(connection state)를 유지할 필요가 없다.
* **작은 헤더 크기**: UDP 헤더는 단 8Byte 로 매우 작아 Network Overhead 가 적다.
* **혼잡 제어 없음**: Network 혼잡 상황을 신경 쓰지 않고 원하는 속도만큼 데이터를 빠르게 쏟아낼 수 있다.

### 주요 활용처
* 데이터 손실에는 어느정도 내성이 있지만, 전송 지연(속도)에는 민감한 **스트리밍 멀티미디어 애플리케이션에 주로 사용**
* `DNS`, `SNMP`, `HTTP/3` 등의 Protol 에서 기반 Transport Layer 로 사용됨
* 만약 UDP를 사용하면서 신뢰성 있는 전송이나 혼잡 제어가 필요하다면 Application Layer 에서 직접 해당 기능을 구현해야 함

### UDP Segment 구조 및 Checksum
> UDP 세그먼트는 크게 8바이트의 헤더와 애플리케이션 데이터(페이로드)로 구성

```text
0                   15 16                  31 (Bits)
+----------------------+----------------------+
|                      |                      |
|   출발지 포트 번호     |   목적지 포트 번호      |
|  (Source Port #)     | (Destination Port #) |
|                      |                      |
+----------------------+----------------------+
|                      |                      |
|         길이          |        체크섬         |
|       (Length)       |      (Checksum)      |
|                      |                      |
+----------------------+----------------------+
|                                             |
|         애플리케이션 데이터 (Payload)          |
|                                             |
+---------------------------------------------+
```

* **헤더 구성 요소**: 출발지 포트 번호, 목적지 포트 번호, 길이, 체크섬 으로 구성됨
* **길이**: 헤더를 포함한 UDP Segment 전체의 길이를 바이트 단위로 나타냄
* **Checksum**: 전송 중에 발생할 수 있는 데이터 오류(비트 반전 등)를 탐지하기 위해 사용
  * **송신자**: UDP Segment 의 내용을 16비트 정수의 연속으로 간주하여 1의 보수 합을 구한 뒤, 이 값을 체크섭 필드에 기록하여 전송
  * **수신자**: 수신된 Segment 에 대해 체크섬을 계산한 후, 헤더에 적힌 Checksum값과 일치하는지 확인


## Quiz

Q. 웹 브라우저를 실행 중인 호스트 A와 스마트폰 앱을 사용 중인 호스트 B가 동시에 동일한 스트리밍 서버 C의 포트 5555번으로 UDP 데이터그램을 전송했습니다. 서버 C의 전송 계층은 이 두 데이터그램을 어떻게 역다중화(Demultiplexing)하여 처리합니까?
* A. UDP는 비연결형이므로 수신 측에서 각각의 데이터그램에 대해 일시적인(ephemeral) 소켓을 새로 생성하여 할당한다.
* B. 목적지 IP와 목적지 포트 번호가 같으므로 출발지 정보와 관계없이 두 데이터그램은 동일한 소켓으로 전달된다.
* C. 출발지 IP 주소가 다르므로 운영체제는 이를 서로 다른 두 개의 소켓으로 분리하여 전달한다.
* D. 두 데이터그램의 목적지 포트가 같아 충돌이 발생하므로, 나중에 도착한 데이터그램은 네트워크 계층에서 폐기된다.
> B, UDP는 비연결형 프로토콜로, 목적지 IP와 목적지 포트 번호만을 확인하여 역다중화를 수행합니다. 따라서 다른 호스트에서 보낸 데이터라도 목적지가 같으면 같은 소켓으로 들어갑니다.

Q. 송신자가 UDP 세그먼트에 대해 16비트 단위로 1의 보수 합(one's complement sum)을 구하여 체크섬(Checksum)을 생성하고 전송했습니다. 수신자가 오류를 '검출하지 못하는(undetected)' 경우는 다음 중 어느 것입니까?
* A. 네트워크 혼잡으로 인해 UDP 세그먼트의 헤더 길이(Length) 필드 값 자체가 손상되는 경우.
* B. 페이로드(데이터) 부분에서 연속된 3개의 비트가 모두 1에서 0으로 반전되는 경우.
* C. 전송 중 단일 비트(single bit) 하나가 0에서 1로 반전되는 경우.
* D. 서로 다른 두 16비트 워드의 같은 자리(column) 비트가 하나는 0에서 1로, 다른 하나는 1에서 0으로 반전되는 경우.
> D

Q. 구글 등에서 개발한 HTTP/3는 기존 TCP 대신 UDP를 기반 전송 계층 프로토콜로 채택했습니다. 하향식 접근(Top-Down Approach) 관점에서, 굳이 신뢰성이 없는 UDP를 선택하고 애플리케이션 계층(QUIC)에 복잡한 신뢰성 로직을 직접 구현한 가장 타당한 아키텍처적 이유는 무엇입니까?
* A.UDP가 기본적으로 패킷 순서 보장(in-order delivery)을 제공하므로, 애플리케이션 계층에서 순서를 맞추는 로직을 제거할 수 있기 때문이다.
* B.TCP의 3-way 핸드셰이크와 내장된 혼잡 제어가 발생시키는 고정된 지연(delay)을 회피하고, 애플리케이션 특성에 맞는 최적화된 신뢰성/흐름 제어를 직접 구현하기 위해서이다.
* C.UDP 헤더에는 시퀀스 번호(Sequence Number) 필드가 매우 크게 잡혀있어, 대용량 비디오 스트리밍 데이터를 더 쉽게 추적할 수 있기 때문이다.
* D.라우터와 같은 네트워크 코어(Network Core) 장비들이 TCP 패킷보다 UDP 패킷을 더 높은 우선순위로 큐잉(Queuing)하고 포워딩하기 때문이다.
> B


Q. 호스트 A에서 동작하는 프로세스 P1과 호스트 B에서 동작하는 프로세스 P2가 각각 서로 다른 UDP 소켓을 통해 서버 C의 프로세스 P3(포트 8080)로 데이터를 전송하려고 합니다. 서버 C가 이 두 IP 데이터그램(세그먼트 포함)을 수신했을 때 일어나는 일로 가장 적절한 것은 무엇입니까?
* A. 서버 C의 운영체제는 출발지 IP 주소가 다르므로 P3 프로세스 내에 두 개의 독립적인 스레드를 생성하여 각각 할당한다.
* B. 서버 C의 전송 계층은 두 세그먼트의 목적지 포트 번호가 8080으로 동일하므로, 출발지 IP나 포트와 상관없이 P3가 연결된 동일한 하나의 소켓으로 두 데이터를 모두 전달한다.
* C. UDP는 비연결형이므로 섞이는 것을 방지하기 위해 나중에 도착한 패킷은 임의의 다른 대기 포트로 역다중화한다.
* D. 목적지 포트가 같아 충돌이 발생하므로 네트워크 계층에서 두 패킷을 병합한 뒤 단일 세그먼트로 만들어 애플리케이션 계층으로 전달한다.
> B, UDP 역다중화는 목적지 포트 번호만 일치하면 출발지 정보(IP, 포트 번호)가 달라도 모두 같은 소켓으로 향합니다. TCP였다면 연결마다 별도의 소켓을 생성했겠지만, UDP는 연결 상태를 유지하지 않는 단순한 형태를 취합니다.

Q. 전송 계층에서 다중화와 역다중화를 수행하기 위해 세그먼트 헤더에 반드시 포함되어야 하는 핵심 필드 두 가지는 무엇입니까?
* A. 출발지 IP 주소, 목적지 IP 주소
* B. 순서 번호(Sequence Number), 확인 응답 번호(ACK Number)
* C. 출발지 포트 번호, 목적지 포트 번호
* D. 헤더 길이, 체크섬(Checksum)
> C, 전송 계층의 다중화/역다중화는 "어떤 프로세스(소켓)가 보냈고, 어떤 프로세스가 받을 것인가"를 식별하는 작업이므로 포트 번호가 핵심적인 역할을 합니다.


Q. UDP 소켓이 어떻게 완벽하게 식별되는가? TCP 소켓의 경우는 어떤가? 두 소켓의 완벽한 식별 간에는 어떤 차이가 있는가?
> * UDP 소켓 식별: 목적지 IP 주소와 목적지 포트 번호, 즉 2-튜플(2-tuple)로 식별
> * TCP 소켓 식별: 출발지 IP 주소, 출발지 포트 번호, 목적지 IP 주소, 목적지 포트 번호, 즉 4-튜플(4-tuple)로 식별
> * 차이점: 이 식별자의 차이 때문에 역다중화 방식이 달라집니다. UDP는 다른 호스트(A, B)에서 보낸 데이터라도 목적지 포트 번호만 같으면 수신 측의 '동일한 하나의 소켓'으로 들어갑니다. 반면, TCP는 출발지 IP나 포트 번호 중 하나라도 다르면 수신 측에서 '완전히 독립된 별개의 소켓'을 생성하여 매핑합니다.

Q. 왜 애플리케이션 개발자들이 TCP보다는 UDP상에서 애플리케이션을 구동하는 것을 선택하는지를 설명하라.
> 애플리케이션 계층에서 데이터 송신 시점과 전송 속도를 더욱 정밀하게 제어하기 위해서입니다.
> TCP는 자체적인 '혼잡 제어(Congestion Control)' 기능이 있어 네트워크가 혼잡하다고 판단되면 강제로 송신 속도를 늦춥니다. 또한 신뢰성을 보장하기 위해 재전송을 수행하므로 지연(Delay)이 발생합니다. 따라서 실시간 화상회의나 온라인 게임처럼 데이터가 약간 유실되더라도 지연 시간 없이 즉시 보내는 것이 훨씬 중요한 애플리케이션에서는 통제권이 앱 개발자에게 있는 UDP를 선호합니다.

Q. 왜 음성과 비디오 트래픽이 오늘날의 인터넷에서는 종종 UDP보다는 TCP상에서 보내질까? (힌트: 우리가 찾고자 하는 답은 TCP의 혼잡 제어 메커니즘과는 상관이 없다.)
> 가장 큰 이유는 방화벽(Firewall)과 NAT 우회(통과) 문제 때문입니다.
> 오늘날 기업이나 가정의 라우터/방화벽은 보안을 이유로 대부분의 UDP 트래픽을 차단하도록 설정되어 있습니다. 반면, 웹 서핑을 위한 TCP 80포트(HTTP)나 443포트(HTTPS)는 항상 열려 있습니다. 따라서 넷플릭스나 유튜브 같은 스트리밍 제공자들은 트래픽이 방화벽에 막히는 것을 피하기 위해, 비디오 데이터를 HTTP(TCP) 스트리밍 형태로 캡슐화하여 전송합니다.

Q. 애플리케이션이 UDP상에서 구동될 때, 신뢰적인 데이터 전송을 누릴 수 있을까? 만약 그렇다면 어떻게 가능한가?
> 네, 가능합니다. 전송 계층(UDP) 자체는 신뢰성을 제공하지 않지만, 그 위에 있는 애플리케이션 계층 코드 내에 신뢰성 보장 로직을 직접 구현하면 됩니다. 애플리케이션 개발자가 확인 응답(ACK) 메시지 교환, 순서 번호 부여, 타이머 설정 및 손실된 패킷에 대한 재전송 기능 등을 자체적으로 프로그래밍하여 넣으면 UDP를 기반으로도 신뢰적인 통신을 달성할 수 있습니다. (구글이 만든 QUIC 프로토콜/HTTP/3가 정확히 이 방식을 사용합니다.)

Q. 호스트 C에 있는 프로세스가 포트 번호 6789를 가진 UDP 소켓을 갖는다고 가정하자. 호스트 A와 호스트 B 모두 각각 목적지 포트 번호 6789를 가진 호스트 C에서 UDP 세그먼트를 보낸다고 가정하자. 이 세그먼트 모두 호스트 C에 있는 같은 소켓으로 전달될 것인가? 그렇다면 호스트 C에 있는 프로세스는 어떻게 두 세그먼트가 다른 두 호스트에서 시작했다고 아는가?

> 같은 소켓으로 전달되는가?: 네, 두 세그먼트 모두 호스트 C의 6789 포트에 바인딩된 동일한 단일 UDP 소켓으로 전달됩니다. (R3 문제의 개념입니다.)

> 어떻게 출처를 구분하는가?: 수신된 UDP 세그먼트는 하위 계층에서 올라올 때 IP 데이터그램에 감싸져 있습니다. 호스트 C의 운영체제는 패킷을 역다중화하여 소켓으로 올릴 때, 단순히 페이로드 데이터만 올려보내는 것이 아니라 IP 데이터그램 헤더에 있는 '출발지 IP 주소'와 UDP 헤더에 있는 '출발지 포트 번호'를 애플리케이션 프로세스에 함께 넘겨줍니다. 프로세스는 이 반환된 정보를 확인하여 누가 보낸 데이터인지 식별하고 구분할 수 있습니다.
