# 운영체제 기능
* 프로세서, 기억장치, 입출력장치, 파일등 자원 관리

* 인터페이스 제공

* 데이터, 자원 관리 및 공유

# 운용 기법
* 일괄처리(Batch Processing) 시스템: 초기 형태
* 다중 프로그래밍(Multiprogramming) 시스템: 여러 프로그램을 메모리에 올려 놓고 CPU가 여러 프로그램을 번갈아 가며 실행
* 시분할(Time Sharing) 시스템: 다중 프로그래밍 시스템에서 발전된 형태, CPU 시간을 분할하여 여러 사용자가 동시에 사용, Round Robin 방식 이라고도 함, 대화식 처리 가능
* 다중 처리(Multiprocessing) 시스템: 여러 CPU, 하나의 메모리 사용, CPU 하나 고장나도 다른 CPU가 대신 처리, 안정성이 좋음
* 실시간 처리(Real-time Processing) 시스템: 정해진 시간 안에 처리해야 하는 시스템, 실시간 처리 시스템은 Hard Real-time, Soft Real-time으로 나뉨, 우주선 발사 시스템 같은곳에 사용
* 다중 모드 처리(Multi-mode Processing) 시스템: 일괄처리, RR, 다중처리, 실시간 시스템 모두 제공
* 분산처리(Distributed Processing) 시스템: 여러 컴퓨터가 연결되어 하나의 시스템처럼 동작, 네트워크로 연결, 각 시스템은 고유의 CPU와 메모리를 가짐

# 스케줄링
* 비선점(Non-preemptive) 스케줄링: 프로세스가 CPU를 할당받으면 CPU를 반환할 때까지 다른 프로세스가 CPU를 강제로 빼앗을 수 없음
* 선점(Preemptive) 스케줄링: 프로세스가 CPU를 할당받아 실행 중에 다른 프로세스가 CPU를 강제로 빼앗을 수 있음

## 비선점 스케줄링
* FCFS(First Come First Served): 먼저 도착한 프로세스가 먼저 CPU를 할당받음
* SJF(Shortest Job First): 실행 시간이 가장 짧은 프로세스부터 CPU를 할당받음
* HRN(Highest Response Ratio Next): SJF의 변형, 대기 시간을 고려하여 우선순위를 정함
    * 우선순위 = (대기시간 + 실행시간) / 실행시간
* 기한부(Deadline): 프로세스의 실행 시간이 정해져 있을 때 사용, 프로세스의 실행 시간이 기한을 넘어가면 제거 or 처음부터 다시 실행

## 선점 스케줄링
* RR(Round Robin): 시분할 시스템에서 사용, 각 프로세스에게 동일한 시간 할당
* SRT(Shortest Remaining Time): SJF의 선점형, 실행 시간이 가장 짧은 프로세스부터 CPU를 할당받음
* 다단계 큐(Multi-level Queue): 프로세스를 여러 개의 큐로 나누어 관리, 각 큐마다 다른 스케줄링 알고리즘 사용
* 다단계 피드백 큐(Multi-level Feedback Queue): 다단계 큐에서 발전된 형태, 프로세스가 다른 큐로 이동 가능

# 데드락
충분조건
* 상호배제(Mutual Exclusion): 자원은 한 번에 한 프로세스만 사용 가능
* 점유대기(Hold and Wait): 프로세스가 자원을 가진 상태에서 다른 자원을 기다림
* 비선점(No Preemption): 자원을 강제로 빼앗을 수 없음
* 환형대기(Circular Wait): 프로세스 간의 자원 요청이 순환 형태로 대기

해결 방법
* 예방(Prevention): 충분조건중 하나를 제거, 자원의 낭비가 심함
* 회피(Avoidance): 발생하면 적절히 피함, 응행원 알리즘(Banker's Algorithm) 사용
* 발견(Detection): 시스템에 데드락이 발생했는지 주기적으로 검사
* 회복(Recovery): 데드락 발생 시 복구, 프로세스 종료, 자원 선점

# 화이트박스 테스트
* 소프트웨어의 내부 구조를 검사하는 테스트
* 모듈의 모든 문장을 한 번 이상 실행
* 모든 경로를 테스트하여 테스트케이스를 설계

## 화이트박스 테스트의 종류
* 기초 경로 검사(Base path Testing): 설계자가 작성한 프로그램의 모든 가능한 경로를 테스트
* 제어 구조 검사(Control Structure Testing): 프로그램의 제어 구조를 검사
    * 조건 검사(Condition Test): 모듈 내의 조건문 검사
    * 루프 검사(Loop Test): 루프를 중점으로 검사
    * 데이터 흐름 검사(Data Flow Test): 변수(데이터의) 흐름을 중점

## 검증 기준
* 문장 검증(Ststement Coverage): 소스코드의 모든 구문이 한번 이상 수행되도록 테스트
* 분기 검증(Branch Coverage): 모든 분기문이 한번 이상 수행되도록 테스트(True, False)
* 조건 검증(Condition Coverage): 조건 내부에 포함된 개별 조건까지 테스트
* 분기/조건 검증(Decision/Condition Coverage): 위 두개 모두 만족

# 블랙박스 테스트
* 소프트웨어의 기능을 검사하는 테스트, 기능 테스트라고도 함
* 사용자의 요구사항 명세를 보면서 테스트
* 주로 구현된 기능을 테스트

## 종류
* 동치 분할 검사(Equivalence Partitioning): 입력 조건을 틀린거, 맞는거 반반 넣어서 테스트
* 경계값 분석 검사(Boundary Value Analysis): 입력 조건이 경계값일시 테스트
* 원인-효과 그래프 검사(Cause-Effect Graphing): 입력 조건과 출력 결과를 그래프로 표현하여 테스트(분석해 테스트)
* 오류 예측 검사(Error Guessing): 경험에 의존하여 테스트
* 비교 검사(Comparison Testing): 여러 버전의 프로그램에 동일한 테스트 자료 제공


# 개발 단계에 따른 애플리케이션 테스트
| 소프트웨어 개발 단계 | 테스트 단계 |
|:---:|:---:|
| 요구 분석 (Requirement Analysis) | 인수 테스트 (Acceptance Testing) |
| 분석 (Analysis) | 시스템 테스트 (System Testing) |
| 설계 (Design) | 통합 테스트 (Integration Testing) |
| 구현 (Code) | 단위 테스트 (Unit Testing) |

* 유닛 테스트: 모듈이나 컴포넌트에 초점을 맞춰 테스트, 단독으로 실행할 수 있는 환경과 데이터 필요
* 통합 테스트: 유닛시스템을 하나로 완성시키는 과정에서의 테스트
* 시스템 테스트: 개발된 소프트웨어를 테스트
* 인수 테스트: 사용자의 요구사항을 총족 하는지 테스트
    * 알파테스트: 개발자가 보는 앞에서 사용자가 테스트
    * 베타테스트: 사용자가 직접 테스트
